# Recommendation Explainability — Axis

**Date:** 2026-07-16
**Method:** Direct code reading and tracing, matching the discipline of `docs/ux/UXStabilizationAudit.md`. Every claim below is grounded in a traced code path with file/line references, not inferred from behavior.
**Status:** Audit + design document only. No production code was modified while producing this document.
**Origin:** UX Stabilization extension brief, Part 3.

**Headline finding:** the extreme percentages users see (+1200%, -1800%, +480%) are not a display-formatting oversight. They're the visible symptom of a deeper structural problem: Axis's "explanation" of a recommendation is generated by a *second, independent, simplified model* that is only loosely connected to the pipeline that actually produced the recommendation — and that second model is then forcibly rescaled to agree with the real number after the fact. When the two disagree enough, the rescaling factor explodes, and every signal's displayed impact explodes with it. Fixing the display (clamping the percentage) would hide the symptom. The section below traces the actual disease.

---

## 1. The Explainability Audit

### 1.1 How a recommendation actually gets its final numbers

There is no single function that computes "today's recommendation." There are at least **three semi-independent layers**, each producing its own volume/intensity modifier, that get composed — not always coherently:

**Layer 1 — Progression × Readiness ceiling** (`lib/exercises/generateWorkout.ts`, `resolveEffectiveAdjustment()`, lines 247–284). Combines the progression engine's `CoachingAdjustment` (adherence- and mastery-driven) with `ReadinessScore` as a conservative ceiling — readiness can only pull volume *down* from what progression wants, never up. Produces a `volumeModifier` applied inside `prescribeExercise()` (line 315): `sets = max(1, round(rawSets * volumeMod))`.

**Layer 2 — Auto-regulation session scaling** (`lib/autoregulation/trainingDecisionEngine.ts` → `computeSessionScaling()`). A *separate* function that independently ingests readiness, recovery debt, burnout risk, fatigue, symptoms, deload status, performance trends, and adherence risk, and produces its own `TrainingDecision.finalVolumeScale`. This is displayed directly in `TrainingDecisionCard.tsx` and is a coherent, bounded number on its own (schema-enforced 0.0–2.0, `lib/data/schema/coreSchemas.ts` line 64).

**Layer 3 — Adaptive pattern multiplier + periodization** (`lib/exercises/generateWorkout.ts`, lines 468–505). A *third* pass: `adaptiveVolumeMultiplier` (personal-pattern learning, ±15%) is applied, then `periodizationStatus.setsOffset` (a flat ±1/±2 sets, from `lib/periodization/goalProfiles.ts` — accumulation +1, deload -2) is applied on top.

**The composition problem:** Layer 2's `finalVolumeScale` — which is *already* a composite of readiness, recovery, burnout, fatigue, symptoms, deload, and adherence — is passed into `generateWorkout()`'s `adherenceRiskScale` parameter (`app/dashboard/page.tsx` line 2520/2733: `runWorkoutPipeline(..., safetyResult?.volumeScale ?? trainingDecision?.finalVolumeScale ?? 1.0)`), which `generateWorkout.ts` applies as **a fourth multiplicative reduction** (line 486–491), on top of Layer 1's readiness ceiling and Layer 3's pattern multiplier — both of which *already* factored in readiness, fatigue, and adherence through their own paths.

**Concretely:** readiness alone can suppress volume via Layer 1's ceiling (`resolveEffectiveAdjustment`), *and* via Layer 2's `finalVolumeScale` (which also reads readiness), *and* that same `finalVolumeScale` gets applied a second time as if it were a distinct "adherence" factor in Layer 3. The same underlying signal is counted more than once, through variable names (`volumeModifier`, `finalVolumeScale`, `adherenceRiskScale`) that don't make the overlap visible to whoever is reading any one of these files in isolation. This is the direct cause of Part 1's "most exercises show 1 set" finding (see `docs/ux/UXStabilizationAudit.md` Part 1 §1) — four compounding reductions, each independently floored at `max(1, ...)`, several of them counting overlapping signals.

### 1.2 How the "explanation" is generated — and why it diverges

`lib/intelligence/recommendationExplanation.ts`, `generateRecommendationExplanation()`, is **not a readout of layers 1–3 above**. It's a fourth, independent, much simpler model: nine hand-tuned linear coefficients against raw signal values (lines 55–101):

```ts
const rawReadiness   = (readinessScore  - 70)   * 0.004;
const rawRecovery    = (recoveryScore   - 70)   * 0.003;
const rawBurnout     = (30 - burnoutRiskScore)  * 0.002;
const rawFatigue     = (50 - fatigueScore)      * 0.0015;
const rawCycle       = cyclePhaseRawImpact(cyclePhase);      // ±0.02 to ±0.05, hardcoded per phase
const rawAdherence   = adherenceScale - 1.0;
const rawUncertainty = uncertaintyModifier - 1.0;
const rawSymptoms    = -(symptomCount * 0.012 + symptomSeverityMean * 0.018);
const rawMomentum    = momentumRawImpact(momentumLevel);     // ±0.02 to ±0.03, hardcoded
```

This model has no knowledge of periodization, adaptive pattern learning, or the actual composition described in §1.1 — it's a plausible-looking *guess* at what probably influenced the day's volume. Because it's a guess, it essentially never sums to the real `finalVolumeScale` that Layers 1–3 actually produced. So the code **forces** it to agree, after the fact (lines 106–111):

```ts
const rawSum      = rawReadiness + rawRecovery + rawBurnout + rawFatigue +
                     rawCycle + rawAdherence + rawUncertainty + rawSymptoms + rawMomentum;
const actualDelta = finalVolumeScale - 1.0;                    // the REAL delta from layers 1–3
const scale        = Math.abs(rawSum) > 0.001 ? actualDelta / rawSum : 1.0;
const pp = (v: number) => Math.round(v * scale * 100);          // every signal's displayed % impact
```

**This is a division whose denominator (`rawSum`) can be arbitrarily close to zero** — the `0.001` guard is far too permissive to prevent instability. `rawSum` approaches zero exactly when the nine hand-picked signals happen to roughly cancel each other out (a completely plausible, common state — e.g. good readiness offset by a rough cycle day). When that happens while the *real* delta (from the actual, more powerful, compounding Layers 1–3) is a typical-sized number like -0.20 to -0.30, `scale` becomes a large multiplier — tens or hundreds — and it is applied identically to **every** signal's individual percentage.

**Worked example, using the real formula above**, showing how a -1200%-class number is produced:

- Suppose `rawSum = 0.0015` (just above the guard) and the real `actualDelta = -0.30` (a 30% reduction — unremarkable, easily reached by Layers 1–3 stacking).
- `scale = -0.30 / 0.0015 = -200`.
- A single moderate raw contribution, say `rawCycle = 0.06` (Ovulatory boost): `pp(0.06) = round(0.06 × -200 × 100) = round(-1200) = -1200%`.
- Every other signal is multiplied by the same `-200` scale simultaneously — so the user sees several wildly-different-looking numbers (+1200%, -1800%, +480%) all appear together, which is exactly what was reported. They are not independent glitches; they are one glitch, applied nine times.

### 1.3 Where the unclamped number reaches the user

`components/intelligence/RecommendationExplanationCard.tsx`, `ImpactRow` (lines 12–40):

```tsx
const bar = Math.min(Math.abs(impact), 20);          // ← bar WIDTH is clamped to 20
...
{positive ? "+" : ""}{impact}%                        // ← but the TEXT is the raw, unclamped value
```

The bar is capped so it never visually overflows its row — but the printed number next to it is not. A user sees a short, unremarkable-looking bar next to the text "+1200%". This inconsistency (visually bounded, numerically unbounded) is itself informative: whoever wrote this component already recognized the value needed a ceiling for the bar and simply didn't apply the same ceiling to the text. `components/intelligence/WhatChangedCard.tsx`'s `VolumeArrow` and the top-level "Volume ±N%" chip in `RecommendationExplanationCard.tsx` (line 64, `deltaPercent`) are **not** affected — those read `finalVolumeScale` directly, which is schema-bounded 0–2, so their percentages are sane. The bug is isolated to the *per-signal breakdown*, not the headline number.

### 1.4 Per-signal contribution summary (as currently modeled)

| Signal | Where it's modeled | Bounded at source? |
|---|---|---|
| Readiness | Layer 1 (ceiling) + Layer 2 (`computeSessionScaling`) + explanation model's `rawReadiness` | Layer 1/2: yes. Explanation model: only after the unstable rescale |
| Recovery | Layer 2 + explanation model's `rawRecovery` | Same pattern |
| Cycle phase | `generateWorkout.ts`'s `PHASE_RPE_DELTA`/`PHASE_NOTES` (RPE only, not volume) + explanation model's hardcoded `cyclePhaseRawImpact()` (invents a volume effect the actual generation pipeline doesn't apply) | The explanation model asserts a cycle-phase *volume* effect that Layer 1–3 don't actually implement — phase only adjusts RPE target in the real pipeline. This is a second, distinct honesty gap: the explanation can claim a driver that didn't mechanically act. |
| Symptom burden | Layer 2 (`symptomSeverityMean`, `symptomCount`) + explanation model | Layer 2: yes |
| Fatigue | Layer 2 (`fatigueEntry.score`/`.zone`) + explanation model's `rawFatigue` | Layer 2: yes |
| Training momentum | Explanation model's `momentumRawImpact()` only — **momentum does not feed Layers 1–3 at all** | Asserted in the explanation with no mechanical counterpart, same gap as cycle phase |
| Adherence risk | Layer 2 + double-counted into Layer 3 (§1.1) + explanation model's `rawAdherence` | Triple-referenced across the pipeline |
| Periodization / deload | Layer 3 only (`setsOffset`, `rpeOffset`) | **Not represented in the explanation model at all** — a real, often-large driver (deload = -2 sets) that the user is never told about when it's the actual cause of a big reduction |

Two distinct classes of problem, both real:
1. **Instability** (§1.2/§1.3): the displayed percentages can mathematically explode.
2. **Dishonesty by omission and invention**: the explanation model asserts effects (cycle phase and momentum on *volume*) the real pipeline doesn't implement, and omits an effect (periodization/deload) that's often the single largest real driver.

## 2. Short-Term Adaptation (daily)

What actually changes day to day, traced to source:

- **Readiness** (`lib/readiness/calculateReadiness.ts`) — computed fresh each day from sleep, HRV-adjacent signals, and recent training load; feeds Layer 1's ceiling and Layer 2's scaling.
- **Symptoms** (`todaySymptomsVal`, logged same-day) — feeds Layer 2 directly, and the movement-preparation layer (`lib/movement/symptomMovementModifiers.ts`) which can substitute exercises, not just scale them.
- **Fatigue** (`lib/autoregulation/fatigueModel.ts`, `computeCurrentFatigueScore()`) — a same-day composite of training load, recovery bank, sleep, stress, symptoms (see `docs/ux/UXStabilizationAudit.md` Part 3.3 for its own maturity-gating work).
- **Recovery** (`lib/recovery/recoveryScore.ts`) — today's retrospective recovery state, feeds Layer 2's `recoveryDebtScore`.
- **Recent training** — `existingVolumeScale` input to Layer 2 already reflects Phase 34/35's rolling adherence/consistency composite.
- **Sleep, nutrition** — sleep quality feeds fatigue (§ above) and readiness; nutrition currently does **not** feed any volume/intensity modifier in Layers 1–3 (it has its own separate `FuelingCard`/`NutritionIntelligenceCard` surface, decoupled from training-volume decisions — worth noting since the brief asks for nutrition to be described as a short-term input, and today it isn't mechanically one).

**How these combine into today's recommendation:** per §1.1, not through one function — through the three-layer composition, with the real risk that overlapping signals (readiness and adherence, specifically) are counted more than once.

## 3. Long-Term Adaptation (weekly / monthly)

- **Calibration** (`lib/adaptive/accuracyCalibration.ts`, Phase 57) — compares predicted vs. actual outcomes over time, feeds `ConfidenceInputs.calibrationFactor`.
- **Verification** (`lib/intelligence/verification/`, Phase 64) — the formal `VerificationRecord`/`VerificationState` lifecycle (`waiting` → `verified`/`insufficient_data`) that scores whether past recommendations worked; feeds `ConfidenceInputs.verificationSuccessRate`.
- **Adherence** — `lib/adherence/consistency.ts`'s `historyDepth`-gated composite (see UX Stabilization Batch 3), rolling into Layer 2's `existingVolumeScale`.
- **Historical performance** — `lib/performance/performancePredictors.ts` and `lib/analytics/performanceTrends.ts` feed `dominantTrend()` in the Training Decision Engine (§1.1 Layer 2), one of the few places long-term data directly changes a short-term decision.
- **Confidence growth** — see §4/§5 below. **Currently this is the weakest link**: confidence is computed comprehensively (`ConfidenceEngine.ts`) but, per §5, is not wired into how *aggressively* Layers 1–3 are allowed to adjust volume. It's tracked; it doesn't yet act.
- **Athlete development** — `lib/progression/` (mastery, volume landmarks, progression targets) is the closest thing to a genuine long-term adaptation model, feeding Layer 1's `CoachingAdjustment`.

## 4. Data Maturity

Axis already has a real maturity model — `lib/intelligence/confidence/ConfidenceCalculator.ts`, `deriveMaturityStage()` — keyed on **completed workouts**, not calendar time:

| Stage (as implemented) | Threshold | Approximate calendar mapping (at ~3–4 sessions/week) |
|---|---|---|
| `getting_started` | 0–10 workouts | **Day 1 – end of Week 2ish** |
| `learning` | 11–25 workouts | **~Week 3 – Week 6** (roughly Month 1) |
| `personalized` | 26–50 workouts | **~Month 2 – Month 3** |
| `highly_personalized` | 51–100 workouts | **~Month 3 – Month 6** |
| `long_term_profile` | 101+ workouts | **Month 6+** |

This is a better anchor than a pure calendar model — a user training 5x/week reaches `personalized` faster than one training 2x/week, which is the right behavior for a system whose confidence should track *evidence volume*, not elapsed time. The brief's Day 1/Week 1/Month 1/Month 3/Month 6 framing maps cleanly onto this existing scale (above) rather than needing a new, parallel calendar-based system.

**Expected behavior at each stage (design recommendation, not yet implemented — see §7):**

- **Day 1 (`getting_started`, 0 workouts):** No adaptive adjustment beyond the deterministic phase/energy-level baseline (`ENERGY_PRESETS` in `generateWorkout.ts` — this part is fine as-is, it's not personalization, it's a starting template). Explanation surface should not claim any personalized driver — "Building your baseline" language, matching the pattern already established for Adherence/Recovery cards in UX Stabilization Batch 3.
- **Week 1–2 (`getting_started`):** Same. Layer 2's auto-regulation can still react to same-day signals (symptoms, sleep) — that's not personalization, that's responding to today's inputs, which is legitimate at any maturity level and should not be gated.
- **Month 1 (`learning`):** Layer 1's progression engine and Layer 3's adaptive pattern multiplier should be allowed to start acting, but at reduced magnitude (§6).
- **Month 3 (`personalized`):** Full progression/pattern-learning magnitude reasonable.
- **Month 6+ (`highly_personalized`/`long_term_profile`):** Full system, including periodization-driven deload swings, which are the largest legitimate single-session changes Axis makes.

## 5. Confidence Scaling

`ConfidenceEngine.ts`, `buildConfidenceProfile()`:

- `compositeScore` (0–1) is a weighted aggregate of six dimensions (`calculateDimensions()` in `ConfidenceCalculator.ts`): calibration, verification, data completeness, personalization maturity, prediction stability, historical context.
- `level` is derived from that score (`scoreToLevel()`, `ConfidenceEngine.ts` lines 15–22): `Insufficient` (<0.15, or <2 available dimensions), `Limited` (0.15–0.35), `Building` (0.35–0.55), `Moderate` (0.55–0.75), `High` (≥0.75).
- **Confirmed by direct trace (`app/dashboard/page.tsx`, grep for `confidenceProfile\.`): the only current consumer of `confidenceProfile` is `<ConfidenceBadge level={confidenceProfile.level} />` (line 2830) — a read-only display chip.** No file in `lib/exercises/`, `lib/autoregulation/`, or `lib/adaptive/` references `confidenceProfile` or `ConfidenceProfile` at all. Confidence is computed thoroughly and then not used to constrain anything. This is precisely the gap the brief names: "the recommendation engine should gradually increase adaptation magnitude as confidence grows... integrate with the existing Confidence Engine rather than bypass it" — today, it's not bypassed so much as never connected in the first place.

## 6. User-Facing Explainability — Proposed Redesign

### 6.1 Replace per-signal percentages with qualitative bands

Rather than printing `explanation.readinessImpact` directly, bucket the magnitude:

| `|impact|` (percentage points, current scale) | Label |
|---|---|
| 0–2 | *(omitted — not shown, matches existing `if (impact === 0) return null` intent)* |
| 3–7 | Small impact |
| 8–15 | Moderate impact |
| 16+ | High impact |

This requires the underlying number to be trustworthy at all — §6.2 is the prerequisite, not a replacement for fixing §1.2's instability. A qualitative label computed from an exploding number is still wrong, just wrong in a way that's harder to notice (a signal that should be "small" could land in "high impact" purely from a `rawSum`-near-zero coincidence).

Proposed layout (matches the brief's example, adapted to the actual signal set from §1.4):

```
Today's Recommendation

Primary factors

High impact
• Reported fatigue

Moderate impact
• Menstrual symptoms

Small impact
• Training momentum
```

Grouped by band, signal order within a band by `Math.abs(impact)` descending (existing sort in `generateRecommendationExplanation()`, line 138, is reusable).

### 6.2 Fix the instability at the source, not just the display

Clamping the display number (e.g. `Math.min(Math.abs(impact), 100)`) would stop the absurd text but would **not** fix the underlying dishonesty — a clamped-to-100% "Menstrual symptoms" driver is still a fabricated number if the real cause was an unrepresented periodization deload (§1.4). Two changes are needed together:

1. **Represent every real driver.** Add periodization/deload as an explicit signal in `generateRecommendationExplanation()`'s input and raw-contribution set (it's readily available — `periodizationStatus.setsOffset`/`.rpeOffset` already exist, they're just not threaded into this function today). Remove or clearly caveat the cycle-phase and momentum "volume" contributions, since — per §1.4 — the real pipeline doesn't apply them to volume today; either wire them in for real (a recommendation-logic change, out of scope for this pass) or stop claiming them as volume drivers in the explanation (a documentation-honesty fix that requires no pipeline change).
2. **Stop rescaling per-signal values to force-match an external total.** The calibration hack (§1.2) exists because the explanation model's own math doesn't naturally sum to the real delta. The sustainable fix is for the explanation model to read from the *actual* layers (§1.1) rather than approximate them independently — i.e., have `generateWorkout.ts` return its own delta decomposition (how much Layer 1 changed things, how much Layer 2, how much Layer 3) instead of a black-box final number that a fourth, disconnected model then has to reverse-engineer. This is the only way to make the explanation *true* rather than merely *bounded*.

### 6.3 Headline summary stays prose, gets more specific

`buildSummary()` (`recommendationExplanation.ts` lines 176–235) already produces reasonable prose ("Today's training volume has been reduced. This is primarily because of ... "). Once §6.2's driver set is honest, this function's output becomes trustworthy by construction — no redesign needed here beyond feeding it accurate inputs.

## 7. Recommendation Constraints — Conservatism Tied to Confidence

**Current state (confirmed, §5): no confidence-based damping exists anywhere in the volume/intensity pipeline.** A brand-new user (`Insufficient` confidence, `getting_started` maturity) can receive the exact same-magnitude swings — including a full periodization deload (-2 sets, §1.1 Layer 3) or a Layer 2 auto-regulation scale-down — as a six-month user with `High` confidence and a fully personalized profile. Nothing in `generateWorkout.ts` or `trainingDecisionEngine.ts` reads `ConfidenceProfile` today.

### Proposed mechanism (design only)

A single new damping factor, applied once, at the same point Layer 3's `adaptiveVolumeMultiplier` is already applied (`generateWorkout.ts` line 471) — **not** a new independent multiplier stacked on top of the existing four (that would repeat exactly the compounding mistake documented in §1.1). Concretely:

```ts
// Proposed — NOT implemented. Illustrative only.
function confidenceDampingFactor(level: ConfidenceLevel): number {
  switch (level) {
    case "Insufficient": return 0.5;   // adjustments are pulled halfway back to neutral (1.0)
    case "Limited":      return 0.6;
    case "Building":     return 0.75;
    case "Moderate":     return 0.9;
    case "High":         return 1.0;   // full magnitude, unchanged from today
  }
}

// Applied to the DEVIATION from neutral, not the multiplier itself, so it
// damps magnitude without changing direction:
const damped = 1.0 + (rawModifier - 1.0) * confidenceDampingFactor(level);
```

Applying the damping to the *deviation from 1.0* (not to the raw multiplier) is the important detail — it means a `Insufficient`-confidence user with a computed `rawModifier` of 0.70 (a 30% cut) gets `1.0 + (0.70 - 1.0) × 0.5 = 0.85` (a 15% cut) — still directionally a reduction, still responsive to today's real signals, just not swinging as far while the system has little evidence its model is calibrated for this specific person. This directly satisfies the brief's requirement: *"Without sufficient data: avoid 40–50% workout reductions. Prefer small, confidence-aware adjustments... gradually increase adaptation magnitude as confidence grows."*

**Where this should — and shouldn't — apply:** dampen Layer 1 (progression ceiling) and Layer 3 (adaptive pattern multiplier + periodization), since both are explicitly *personalization* mechanisms that are only as good as the history behind them. Layer 2 (auto-regulation session scaling) responds to *today's* signals (symptoms, acute fatigue, readiness) — these aren't personalization claims, they're direct responses to same-day state, and a brand-new user reporting symptoms today deserves the same responsiveness as anyone else. Damping Layer 2 would be a safety regression, not a trust improvement; this distinction is why the brief's constraint is framed as "adaptive recommendations" specifically, not all recommendations.

**Integration point:** this reads `ConfidenceProfile.level`, already computed every pipeline run (`app/dashboard/page.tsx`, `buildConfidenceProfile()`) — it needs to be *passed into* `generateWorkout()` as a new optional input (alongside the existing `adaptiveVolumeMultiplier`) rather than computed independently inside `generateWorkout.ts`, keeping the Confidence Engine as the single source of truth for "how much do we trust this."

## 8. Single Source of Truth Architecture (Required)

**Status:** Architecture requirement, documented per explicit instruction — not yet implemented. This section formalizes the target architecture that §§1–7 above have been building the case for; it doesn't introduce new findings so much as name the principle that Findings E1–E5 are all instances of violating.

### 9.1 The principle

The Adaptive Intelligence pipeline is the single source of truth for every recommendation, its magnitude, its confidence, and its explanation. The frontend's job is presentation only. Concretely, the frontend must never:

- independently calculate a recommendation adjustment,
- estimate a contributing factor,
- infer recommendation reasoning,
- rescale an intermediate value,
- normalize an output separately from where it was produced,
- or reconstruct an explanation from partial information.

**Every violation already found in this audit is a violation of this exact principle, not a separate class of bug:**

| Finding | How it violates single-source-of-truth |
|---|---|
| E1 (§1.2) | `recommendationExplanation.ts` **reconstructs** an explanation from partial information (nine hand-picked raw signals) instead of reading the real pipeline's own decomposition |
| E2/E3 (§1.4) | The reconstruction **invents** contributors (cycle phase, momentum) the real pipeline doesn't implement, and **omits** one (periodization) it does |
| E4 (§1.1) | The pipeline itself has no single source internally — three semi-independent layers each produce their own volume modifier, composed by parameter-passing convention (a value literally named for one thing carrying a different thing, §1.1's Layer 2→3 handoff) rather than by one coordinated calculation |
| E5 (§5) | Confidence is computed once, centrally, correctly — then **not propagated** anywhere; it exists only as a value attached after the fact to a UI badge, which is a milder version of the same "computed in isolation from where it's used" problem |

And one more, found while drafting this section — **the same pattern exists in this audit's own recently-shipped work**, not just in older code:

| New finding | Detail |
|---|---|
| E6 | `lib/intelligence/confidence/ConfidenceTypes.ts` (Phase 67, pre-existing) already defines a comprehensive `MaturityStage` (`"getting_started"` → `"long_term_profile"`, workout-count-thresholded, with human-readable `MATURITY_STAGES` metadata) as part of `ConfidenceProfile`. UX Stabilization Batch 3 (this same audit thread, already shipped) independently introduced a **second, differently-shaped** `MaturityStage` type (`lib/intelligence/dataMaturity.ts` — `"locked"\|"building"\|"ready"`, 7-check-in-thresholded) with no awareness of the first one. Four UI components (`AdherenceCard.tsx`, `FatigueCard.tsx`, `PerformanceHubCard.tsx`, `LockedInsight.tsx`) call `getMaturityStage(entryCount)` directly from the component layer, each supplying its own `entryCount` (`consistency.historyDepth`, `capacity.dataPoints`, `fatigueHistoryDepth` — a mix of per-domain history depths, not the engine's own `ConfidenceProfile.maturityStage`). This is the frontend determining athlete maturity, which §8.4 below explicitly prohibits — and it's proof this isn't a hypothetical risk being guarded against preemptively; it's already happened once in this project, three audit passes into the same stabilization effort. |

### 9.2 Recommendation Pipeline — structured output

Grounding the brief's illustrative shape in what already exists rather than inventing types from scratch:

```ts
interface Recommendation {
  // Existing today, unchanged in kind — types/recommendation.ts DailyRecommendation
  training:   TrainingRecommendation;
  nutrition:  NutritionRecommendation;
  recovery:   RecoveryRecommendation;

  // Existing today (ConfidenceProfile, ConfidenceTypes.ts) — travels WITH the
  // recommendation now, not attached separately after the fact (§8.4)
  confidence: ConfidenceProfile;

  // Existing today (MaturityStage + MATURITY_STAGES, ConfidenceTypes.ts) —
  // reused as-is; lib/intelligence/dataMaturity.ts's parallel type (§8.1, E6)
  // is retired, not kept alongside it
  dataMaturity: MaturityStageInfo;

  // NEW — replaces the three-layer, parameter-passing composition in
  // generateWorkout.ts/trainingDecisionEngine.ts (§1.1) with one coordinated
  // calculation that returns its own decomposition, so nothing downstream
  // has to reconstruct it
  volumeAdjustment:    AdjustmentResult;
  intensityAdjustment: AdjustmentResult;

  // NEW — replaces recommendationExplanation.ts's independent nine-signal
  // reconstruction (§1.2) with a direct readout of what
  // AdjustmentResult.contributors above already computed
  contributors: Contributor[];

  summary:   string;   // today's buildSummary() output — becomes a pure
                        // template over `contributors`, not a second model
  rationale: string;
  warnings:  string[];
}

interface AdjustmentResult {
  finalMultiplier: number;         // bounded, schema-enforced as today (coreSchemas.ts)
  contributors:    Contributor[];  // every signal that mechanically acted, and only those
}

interface Contributor {
  factor:      string;   // "readiness" | "recovery" | "fatigue" | "symptoms" |
                          // "periodization" | "adherence_risk" | ... — a closed,
                          // real set, not whatever nine signals happened to be
                          // hand-picked when recommendationExplanation.ts was written
  impact:      number;   // this factor's actual, mechanically-applied contribution
                          // to finalMultiplier — not independently estimated and
                          // rescaled to match (§1.2)
  magnitude:   "high" | "moderate" | "small";  // pre-bucketed by the engine (§6.1),
                          // not derived by the UI from a raw percentage
  confidence:  ConfidenceLevel;  // this specific factor's own confidence — e.g.
                          // "periodization" is High confidence (deterministic
                          // schedule), "adaptive pattern learning" might be Low
                          // for a new user — a single top-level confidence
                          // number can't represent this
  explanation: string;   // human-readable, generated once, here, not
                          // reconstructed by a UI component
}
```

The key structural change from today: `contributors` is populated by summing what the coordinated calculation (§8.1, fixing E4) *actually did*, field by field, as it does it — not computed twice (once for real, once for display) and reconciled after the fact. This is what makes E1–E3 structurally impossible rather than merely fixed for now: there's no second model left to drift out of sync with the first.

### 9.3 Explainability rule, made mechanical

"Every displayed factor must have a corresponding calculation inside the recommendation pipeline" is enforceable, not just a style guideline, once §8.2's shape exists: a `Contributor` can only exist in the array if the coordinated calculation actually produced it. The failure mode this closes off is exactly E2/E3 — there is no longer a code path where a UI component (or a second explanation-generating function) can add a contributor that wasn't mechanically real, because contributors are a *byproduct* of the real calculation, not an independent description of it.

### 9.4 Confidence as a first-class, traveling property

Per the brief, confidence should influence recommendation magnitude, adaptation limits, explanation wording, feature availability, and data maturity — and should travel through the pipeline, not be attached afterward. Concretely, this means `ConfidenceProfile` becomes an **input** to the coordinated calculation in §8.2 (specifically to the damping mechanism already designed in §7), not a value computed independently and only consumed by a display badge as it is today (E5). The direction of data flow inverts: today, confidence is computed *after* the recommendation, for display; under this architecture, confidence is computed *before* (or alongside, as an early stage) the recommendation, because the recommendation's own magnitude depends on it.

### 9.5 Data maturity, emitted not inferred

Per §8.1's E6 finding, this requires consolidating two existing systems, not building a third:

1. Retire `lib/intelligence/dataMaturity.ts`'s standalone `MaturityStage`/`getMaturityStage()`. Its actual useful logic (a threshold check) migrates into each domain engine that currently calls it — `computeConsistencyScore()`, `computeRecoveryCapacity()`, `computeCurrentFatigueScore()`, `computeTrainingQuality()` — each of which should attach its *own* maturity/confidence signal to its own return value (several already do this in their own vocabulary — `RecoveryCapacity.confidence`, `LifestyleBurnoutReport.dataMaturity` — the migration is about making every engine do it, and do it consistently, not introducing the concept fresh).
2. `AdherenceCard.tsx`, `FatigueCard.tsx`, `PerformanceHubCard.tsx`, `LockedInsight.tsx` stop calling `getMaturityStage(entryCount)` themselves (§8.1's E6 violation) and instead read the maturity field the engine already attached to the object they're already rendering (`consistency.maturityStage`, `capacity.maturityStage`, etc.) — a prop-shape change, not a new capability.
3. `ConfidenceProfile.maturityStage`/`MATURITY_STAGES` (Phase 67, workout-count-based) becomes the one *account-level* maturity signal (used for "is this athlete new to Axis at all"); the per-domain engine maturity fields from step 1 become the *domain-level* signals ("is there enough recovery data specifically" vs. "is there enough consistency data specifically") — these are legitimately different questions and both are needed, but both must come from an engine, never from a UI component doing its own threshold check on a raw count it was hand-fed.
4. The engine additionally exposes, per the brief: remaining observations until the next stage (`MaturityStageInfo.workoutsToNextStage` already has a field for this, currently unpopulated — `ConfidenceTypes.ts` line 48 shows `workoutsToNextStage: null` hardcoded in every entry of `MATURITY_STAGES`, itself a small existing gap worth closing in the same pass), and unlocked/locked capabilities (today implicit in which UI cards choose to render `LockedInsight` — under this architecture, that choice moves to the engine emitting an explicit capability list, and `LockedInsight`'s job shrinks to "render whatever the engine says is locked," nothing more).

### 9.6 UI responsibility

Per the brief, the UI becomes a pure renderer: displaying the `Recommendation` object's fields, rendering `contributors[]` as cards, rendering `confidence`/`dataMaturity` as already-computed badges and locks, rendering educational copy that's either static or engine-supplied — never computing a value that changes what's being communicated. `RecommendationExplanationCard.tsx`'s bar-width clamp (`Math.min(Math.abs(impact), 20)`, §1.3) is a small existing example of the frontier this rule draws: clamping a *rendering* concern (a bar can't visually exceed its row) is fine; the fact that the underlying `impact` number itself needed clamping (§1.2/E1) is not — under this architecture, `Contributor.magnitude` arrives pre-bucketed (§8.2), so there's no raw number left for a UI component to almost-correctly clamp in the first place.

### 9.7 Long-term goal

A deterministic, inspectable pipeline — every `Contributor` traceable from athlete input through the coordinated calculation (§8.2) to the final `Recommendation` — is the natural end state of fixing E1–E6 together rather than individually. Fixing E1 (the display bug) alone would have been a patch. Fixing E1–E6 together, in the shape this section specifies, removes the conditions that produced E1 from existing at all. This is the case for sequencing this as its own reviewed architecture batch (§8.8) rather than a collection of independent point-fixes.

### 9.8 Relationship to the existing Batch 8 plan

`docs/ux/UXStabilizationAudit.md`'s Batch 8 already scoped "decomposition-based explanation model, qualitative-band display redesign, confidence-aware damping" — that scope is a subset of §8.2–§8.6 above and is superseded by this fuller specification (Batch 8 is revised to reference this section directly rather than duplicate it — see that document). The sequencing recommendation is unchanged: this should not start before Batch 5 (the compounding-reduction fix, §1.1/E4) ships and stabilizes, since §8.2's `AdjustmentResult.contributors` needs a *stable* coordinated calculation to decompose — building the decomposition against a pipeline that's simultaneously being corrected means doing the work twice.
